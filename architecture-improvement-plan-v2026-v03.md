# Architecture Improvement Plan (v2026-v03)

## Частина 1: Комплексний аудит коду

Оцінка поточного стану кодової бази додатку Slovko за критеріями Svelte 5 та сучасних архітектурних патернів.

### 1. Управління Станом (State & Reactivity)
*Фундамент: SSoT + Derived over Sync*

* [ ] **Single Source of Truth (SSoT): 70/100**
  * *Аналіз:* Дані здебільшого зберігаються організовано (`gameState.svelte.ts`), але подекуди є ризик розсинхрону через дублювання у локальних змінних компонентів або ручну синхронізацію.
* [ ] **Derived Discipline: 80/100**
  * *Аналіз:* Використовуються `$derived.by` (наприклад, для `accuracy`, `wordsPerMinute`), але все ще є місця, де стан примусово оновлюється в межах ручних методів чи ефектів.
* [ ] **Runes Usage: 90/100**
  * *Аналіз:* Сховища та компоненти активно використовують руни Svelte 5 (`$state`, `$derived`). Класичний API Svelte 4 практично не зустрічається.
* [ ] **URL as State: 10/100**
  * *Аналіз:* Стан (фільтри, активні таби, режим гри) переважно зберігається у пам'яті (`settingsStore` / `let` змінні). При натисканні F5 або шерингу посилання, користувач втрачає свій контекст і бачить дефолтний екран. 

### 2. Архітектура та Потік Даних (Architecture)
*Фундамент: UDF + Separation of Concerns + Server-First*

* [ ] **Server-First (SvelteKit): 0/100**
  * *Аналіз:* Проєкт працює як SPA (Static Site/SPA) з `adapter-static`. Уся бізнес-логіка, запити до API/Firebase та валідація виконуються повністю на клієнті (`+page.server.ts` не використовується).
* [ ] **Type Safety (Zod/Valibot): 50/100**
  * *Аналіз:* Бібліотеку `Zod` додано в бекенд/пакети і частково використовується (`schemas`), але в багатьох сервісах валідація даних з LocalStorage або зовнішніх джерел залишається неявною (просто типізація TypeScript без runtime-схем).
* [ ] **Unidirectional Flow: 70/100**
  * *Аналіз:* Переважно дотримується завдяки поділу на Store/Controller, проте є використання двосторонніх прив'язок (`bind:value`) у формах/модалках, що ускладнює потік даних. 

### 3. Чистота та Побічні Ефекти (Side Effects)
*Фундамент: Purity + Effect Discipline*

* [ ] **Effect Isolation: 60/100**
  * *Аналіз:* Є `$effect`, які безпосередньо взаємодіють з Svelte-станом (наприклад, моніторять `settingsStore.value.mode` і роблять ініціалізацію гри). Змішано синхронізацію з DOM/API і бізнес-логіку.
* [ ] **No State in Effects: 60/100**
  * *Аналіз:* Є ризик потенційних зациклень через те, що деякі ефекти змінюють змінні або викликають методи (`initGame()`), які можуть мутувати реактивний стан.
* [ ] **Cleanup Function: 30/100**
  * *Аналіз:* Майже всі блоки `$effect()` не повертають closure для очищення таймаутів, інтервалів або підписок. (Проте добре працює `<svelte:window>`).

### 4. Компоненти та UI (Component Design)
*Фундамент: Composition + DRY + KISS*

* [ ] **Snippets over Props: 0/100**
  * *Аналіз:* Svelte 5 `{#snippet}` не використовуються взагалі. Замість цього використовуються стандратні слоти або товсті компоненти з безліччю конфігураційних boolean пропсів.
* [ ] **Headless Logic: 90/100**
  * *Аналіз:* Чудове виділення логіки. `gameController.ts` та `gameState.svelte.ts` концентрують в собі бізнес-логіку, залишаючи `GameBoard.svelte` переважно відповідальним за рендер (Presentation).
* [ ] **Locality of Behavior: 90/100**
  * *Аналіз:* Обробка подій (`onclick`, `onpointerdown` тощо) майже завжди знаходиться колоковано прямо в розмітці UI. 

### 5. Продуктивність та Стабільність (Performance & Stability)
*Фундамент: Performance + Error Handling*

* [ ] **Keyed Blocks: 100/100**
  * *Аналіз:* Цикли правильно використовують прив'язку за ключем (`{#key card.id}` в GameBoard та `#each` ключі використовуються скрізь). Стан скидається як треба.
* [ ] **Error Boundaries: 85/100**
  * *Аналіз:* Є кастомний `ErrorBoundary.svelte` та fallback екрани (`+error.svelte`). Нативний `<svelte:boundary>` ще не використовується, але результат досягнуто. 
* [ ] **Lazy Loading: 40/100**
  * *Аналіз:* Асети завантажуються разом. Ледаче завантаження важких модулів через `await import()` не застосовується систематично.

### 6. Безпека та Доступність (Security & A11y)
*Фундамент: Safety*

* [ ] **HTML Injection: 100/100**
  * *Аналіз:* Використання `{@html}` не виявлено. XSS безпека на гарному рівні.
* [ ] **Interactive Elements: 75/100**
  * *Аналіз:* Багато `onclick` вішаються на неінтерактивні елементи (наприклад, `<section role="main">` або `<div>`). Додаються ігнори лінтера (`svelte-ignore a11y_click_events...`), замість того щоб використовувати семантичні `<button>` або повноцінно обробляти клавіатуру.

## Частина 2: План Покращень (Пріоритезований Список Проблем)

Ціль покращень: спростити розуміння коду агентом/розробником, полегшити масштабування та внесення змін без "милиць".

| Статус | Важливість | Задача | Опис та кроки вирішення |
| :---: | :---: | :--- | :--- |
| [ ] | **95/100** | **URL as State (Маршрутизація стану)** | Перенести стан табів, відкритих модалок (наприклад, профілю), виділених фільтрів та режимів гри (`settingsStore.value.mode`) у **URL Search Params** `?mode=playlists&tab=mistakes`. Це дозволить зберігати стан при F5 та покращить незалежність компонентів. |
| [ ] | **85/100** | **Очищення та Ізоляція `$effect`** | Провести ревізію всіх `$effect`. Відв'язати їх від ініціалізації іншого стейту. **Rule of thumb:** `$effect` має використовуватися ВИНЯТКОВО для синхронізації з DOM/LocalStorage і кожен має повертати `() => cleanup()`. |
| [ ] | **80/100** | **Перехід на Svelte 5 Snippets** | Замінити логіку старих `slot` та складних пропсів (типу `showIcon`, `showTranscription`) у компонентах типу `WordCard` та модалках на `{#snippet}`. Це суттєво зменшить кількість коду і зробить композицію прозорою для AI і людини. |
| [ ] | **75/100** | **Zod-валідація всього I/O (Input/Output)** | Написати схеми Zod для всього, що приходить з Local Storage, API або Firebase. Безпосередньо парсити ці дані через `schema.parse()`, щоб гарантувати Type Safety під час виконання (Runtime). |
| [ ] | **60/100** | **Усунення `bind:` (Unidirectional Data Flow)** | Знайти і переписати використання двосторонніх прив'язок `bind:value` на `value={val} oninput={...}`, особливо при роботі зі складним глобальним станом (напр. в налаштуваннях). Дані повинні "текти" лише згори-вниз. |
| [ ] | **55/100** | **Поліпшення Доступності (A11y)** | Видалити всі `svelte-ignore a11y_no_noninteractive_element_interactions`. Замінити клікабельні `div` на `<button>` (можна зі скиданням стилів) або додати відповідні `onkeydown` обробники. AI моделям та скрінрідерам легше орієнтуватись на семантичному HTML. |
| [ ] | **20/100** | **Впровадження `<svelte:boundary>`** | Замінити самописний `ErrorBoundary` на нативний `{#boundary}` від Svelte 5, що значно спростить код перехоплення помилок дочірніх компонентів. |
| [ ] | **10/100** | **Lazy-Loading важких складових** | Перевести важкі модалки або рідко використовувані графічні елементи в ліниве завантаження (`{#await import()}`). |
